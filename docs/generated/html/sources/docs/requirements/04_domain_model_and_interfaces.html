477 ## 4. サーバー最小API契約
478  
479 ### 4.1 `POST /sync/push`
480 - 目的: outbox mutation をクラウドへ反映
481 - 入力:
482 ```json
483 {
484 "deviceId": "macbook-main",
485 "syncMode": "delta",
486 "baseCloudRevision": 100,
487 "mutations": [
488 {
489 "mutationId": "mut_01J...",
490 "entityType": "record",
491 "entityId": "rec_01H...",
492 "operation": "patch",
493 "patch_path": "payload.exit_site_photo",
494 "payload": {
495 "exit_site_photo": {
496 "photo_id": "pho_exit_01H...",
497 "captured_at": "2026-02-09T21:15:00+09:00",
498 "uploaded_at": "2026-02-09T21:16:00+09:00",
499 "source_device_id": "iphone-main"
500 }
501 }
502 }
503 ]
504 }
505 ```
506 - 出力:
507 ```json
508 {
509 "acceptedMutations": ["mut_01J..."],
510 "rejectedMutations": [],
511 "newCloudRevision": 102,
512 "reseedApplied": false
513 }
514 ```
515 - 契約:
516 - `syncMode=delta`: 通常差分同期
517 - `syncMode=full_reseed`: クラウド欠損時にローカル全量を再シード
518  
519 ### 4.2 `POST /sync/pull`
520 - 目的: 指定revision以降の差分取得
521 - 入力:
522 ```json
523 {
524 "deviceId": "iphone-main",
525 "knownCloudRevision": 100,
526 "knownDayRevisions": {
527 "2026-02-09": "sha256:old"
528 }
529 }
530 ```
531 - 出力:
532 ```json
533 {
534 "cloudState": "ok",
535 "cloudRevision": 102,
536 "manifestDiff": {},
537 "dayBundles": [],
538 "photoRefs": [
539 {
540 "photoId": "pho_exit_01H...",
541 "blobKey": "photos/pho_exit_01H....jpg",
542 "photo_kind": "exit_site",
543 "sizeBytes": 202114,
544 "sha256": "7af1...",
545 "updatedAt": "2026-02-09T21:16:00+09:00"
546 }
547 ]
548 }
549 ```
550 - 契約:
551 - `cloudState=ok`: 通常差分応答
552 - `cloudState=missing`: クラウド側の同期基盤が欠損している状態。クライアントは `syncMode=full_reseed` で再シードを実行
553 - `cloudState=missing` の判定条件（いずれか該当で返却）:
554 - `index.json` が存在しない
555 - `index.json` の必須フィールド（`cloudRevision`, `dayRefs`, `photoRefs`, `tombstones`, `integrity`）が欠落または不正
556 - `index.json` が参照する `days/*.json` の必須オブジェクトが1件でも欠損
557 - `index.json` が参照する `photos/*` が1件でも欠損
558 - `integrity` 検証が失敗し、整合性を回復できない
559  
560 ## 5. 競合解決契約（固定）
561 - 解決単位: エンティティ単位
562 - ルール: `(updatedAt, updatedByDeviceId, mutationId)` 降順で勝者決定
563 - `session_summary.payload.exit_site_photo` はフィールド単位LWWで解決し、同一record内の他フィールドは上書きしません
564 - `record` への出口部写真更新は `patch_path=payload.exit_site_photo` の部分パッチで反映します
565 - tombstone: 削除優先なし、同一ルールで勝敗決定
566 - 競合UI: ユーザー向け競合件数表示・詳細表示は行いません（内部適用のみ）
