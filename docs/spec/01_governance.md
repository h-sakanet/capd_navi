# 01. Governance

## 1. 目的
`docs/spec` の運用ルールを固定し、実装時の憶測を排除します。

## 2. 文書優先順位
1. `20_screens/SCR-*.md`
2. `30_capabilities/CAP-*.md`
3. `40_contracts/*`
4. `10_journeys/*.md`
5. `45_nfr/*`
6. `50_quality/*`

## 3. 重複禁止ルール
- 同一仕様を複数ファイルに重複記述しません。
- Journey は UI詳細を再定義しません。
- Screen は横断アルゴリズムを再定義しません。

## 4. 変更ルール
- 仕様変更時は、一次仕様を先に更新します。
- 参照先リンクの更新を同一PRで実施します。
- 変更理由は `90_decisions` に記録します。

## 5. 実装原則
- 「書いていない挙動は実装しない」を原則とします。
- 不明点は `TBD` として明示し、実装前に解消します。

## 6. 文書品質ゲート
- すべての `SCR-*` に GWT を記載します。
- すべての `JRN-*` に参照画面リンクを記載します。
- すべての `CAP-*` に失敗モードと回復方針を記載します。
